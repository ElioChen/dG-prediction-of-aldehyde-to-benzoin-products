### 4. B4/B5重跑:发现一个良性的指标变动,已排查(不是bug)

给B4(`train_gnn_bde.py`)/B5(`train_bondnet_bde.py`)补了`--pred-out`重跑时,醛侧B4结果从
旧值(n=213544, MAE=2.014, R²=0.775)变成了(n=209340, MAE=1.604, R²=0.830)——**主动验证**
排查后确认:两次跑的输入文件(`aldehydes_all.csv`/`aldehydes_bdfe_gxtb_descriptors.csv`)
mtime均早于两次跑,文件本身没变;脚本逻辑除了新增`--pred-out`输出块外无改动。最合理的解释
是旧结果产生于`qc.py`(MAD-based稳健离群点过滤)被抽成共享模块**之前**——当时用的是更宽松
的临时[20,200]窗口过滤,而当前脚本`from qc import qc_filter`用的是重构后更严格的过滤,少
了约4200行(约2%),换来了指标提升。**结论**:新数值(1.604/0.830)是当前代码路径下可信的
数字,应作为B4醛侧的最新基准;旧的2.014/0.775已过时,后续排名以新值为准。B5及B4产物侧的
重跑结果待跑完后一并核实是否有同样的偏移。

### 5. DFT仲裁n=100:n=24的"g-xTB明显更可信"结论没有复现,且发现一个方法学限定

详见memory `dft-arbitration-result`(已更新)。要点:
- n=100下,g-xTB vs ALFABET 的相关系数r差距经bootstrap检验**不显著**(n=24时看似g-xTB
  明显赢,现在是真正的五五开)。
- 但绝对误差(MAE)差距**显著**:g-xTB对本项目自己的DFT参考MAE=14.68,ALFABET=10.05,
  bootstrap 95%CI[3.33, 5.95]kcal/mol,方向和n=24印象相反。
- **主动验证几何confound(用户在下一次对话中问到的点)**:排查后确认g-xTB和本次DFT参考
  几何层级完全一致(都在GFN2-xTB几何上,母体沿用`dft_sp_funnelv3`,片段两边都用同一个
  `_run_xtb_opt`),所以g-xTB vs 本项目DFT的~14kcal差距是干净的电子结构方法差异,不是
  几何伪影,原结论(可能是标签噪声下限)站得住。
- **但ALFABET那条腿不干净**:ALFABET训练标签(BDE-db2)是M06-2X/def2-TZVP、DFT级别几何
  优化、大概率气相,而本次DFT参考是r2SCAN-3c/CPCM(DMSO)/GFN2-xTB几何——三处口径同时不同。
  **ALFABET"MAE更低"不能解读为"更接近真值",只能说它离我们这个specific定义的DFT数字更近**。
  这是`BDE_prediction.md`开篇就提醒过的"不能跨oracle直接比较"在仲裁pilot里的又一次重演。
- **可信的结论只有**:g-xTB相对本项目自己的DFT定义有~14kcal系统偏差,这个偏差是"标签噪声
  下限"假说的更有力证据。ALFABET维持"训练域外、方差趋零"的既有判断,不因这次MAE数字翻案。

## 〇-3、2026-07-16 会话进展:B6 确立为双侧冠军 + B4/B5/B6 三模型集成 + 官能团误差定位

### 1. B6(GNN+局部3D描述符混合)是醛/产物两侧的单模型冠军,幅度可观(非边际)

全库冷启动测试集(`molecule_cold_split`,seed=0,test_frac=0.15)标准 full-test 指标:

| 模型 | 醛 MAE / R² / ρ | 产物 MAE / R² / ρ |
|---|---|---|
| B4 D-MPNN(纯2D图) | 1.604 / 0.830 / 0.940 | 3.641 / 0.834 / 0.918 |
| B5 BonDNet-style | 1.923 / 0.802 / 0.926 | 3.689 / 0.846 / 0.923 |
| **B6 GNN+3D混合** | **1.097 / 0.900 / 0.966** | **2.124 / 0.919 / 0.963** |

B6 相对次优(醛侧 B4、产物侧 B4)的 MAE 降幅:醛侧 −32%(1.604→1.097)、产物侧 −42%
(3.641→2.124),是碾压级差距而非边际提升。**这直接印证了 〇-2 第6节 B6 的设计动机**:
把 H-SPOC 已证明有效的局部电子结构信号(Fukui/WBO/xtb电子结构描述符)通过主 dG 模型验证
过的 `x_d` 融合模式喂进 GNN,零新增计算(两边输入都已在磁盘),换来大幅提升。**B6 应作为
BDE 预测的当前推荐模型。**

### 2. B4+B5+B6 三模型集成:小但真实的额外收益,权重集中在 B6

`pipeline/bde/analyze_b456_ensemble_residuals.py`(把原 2-way 脚本泛化成 N 模型单纯形网格
搜索,step=0.1;权重在测试集随机一半上网格搜索、在另一半上报告,避免同集重拟合乐观偏差)。
共享测试集(三模型测试集按 id 求交)——醛 n=14302、产物 n=7867:

| | 醛侧 MAE(留出半) | 产物侧 MAE(留出半) |
|---|---|---|
| 最优混合权重 | B4/B5/B6 = 0.1/0.1/0.8 | 0.0/0.1/0.9 |
| **混合 MAE / R²** | **1.027 / 0.908** | **2.423 / 0.903** |
| B6 单独(同留出半) | 1.051 | (≈2.44,见 json) |
| 朴素等权 1/3 | 1.239 | 3.046 |

结论:最优混合把 80-90% 权重压在 B6 上,再掺入少量 B4/B5,能在 B6 单独基础上再挤出约 2%
MAE(醛 1.051→1.027),与主 dG 模型 GNN+表格堆叠"−0.051 MAE 小但真实"的模式一致——集成
有真实增益但幅度小,B6 单模型已经拿走绝大部分收益。**朴素等权反而明显更差**(误差被 B5
拖累),说明加权是必要的。

### 5. 交叉微调用更大的 round5 表重跑(承 〇-2 第3节):C>A>B>zero-shot 排名稳健复现

`train_cross_finetune_bde.py` 现支持 `--cross-table`(csv 或 parquet)。用隔壁 dG 会话产出

描述符(H-SPOC/XGB,**不是** B6 GNN):

| 方案 | round3(train=2322)MAE / R² | **round5(train=10107)MAE / R²** |
|---|---|---|
| homo 零样本 | 8.01 / 0.339 | 7.24 / 0.453 |
| A. 仅交叉训练 | 7.48 / 0.420 | 6.22 / 0.575 |
| B. homo+交叉朴素合并 | 7.86 / 0.360 | 6.83 / 0.504 |
| **C. homo 预训练+交叉微调** | **7.22 / 0.447** | **6.11 / 0.585** |

**注意**:round3 与 round5 的测试集是各自表上重新做冷启动划分得到的不同分子子集,绝对 MAE
不能跨列直接比;可比的是**每一列内部的排名**和**趋势**。结论:

- **C > A > B > zero-shot 的排名在 4.3× 更多交叉数据下稳健复现**。再次确认 〇-2 第3节的两个

- **数据规模效应清晰**:交叉训练数据 4.3× 后,所有方案 R² 普涨约 0.13-0.16;且 **C 相对 A 的
  边际优势收窄**(round3:C−A = +0.027 R²;round5:+0.010)——完全符合"微调的优势在交叉数据
  稀缺时最大,数据充足后仅交叉训练自己就能追上"的预期。这是一个干净、可解释的 scaling 结果。
- **推论**:如果继续积累交叉数据,单纯"仅交叉训练"(A)会逐渐够用,微调(C)的相对价值下降;
  但在当前数据量,C 仍是最优,且 R²=0.585 已明显好于 round3 的 0.447。

## 〇-5、MLIP路线(`BDE_prediction.md`第六节)零样本探路——负面结果,不建议现在投入

**第三个可借鉴的信号(留意，暂不必立刻行动)**：隔壁的不确定性校准检查发现，主动学习用的

(预测的acq_score排序和实际的\|Delta\|是否真的相关)，而不是默认相信这个信号有效。

**重要修正(同一天晚些时候)**：隔壁自己那条scaffold-split结论后来被自己推翻了——v1
(`scaffold_split_cv_cross.py`)的`scaffold_n_test`(23000-25000)和`random_n_test`(3776)
根本不是同一个规模，是个测试集大小不匹配的bug；v2重新跑(同一天，`scaffold_split_check_

scaffold-disjoint的R²(~0.77)反而略高于随机划分(~0.70-0.76)，也就是说隔壁的交叉dG模型
其实**没有**scaffold泛化问题。

**这是否推翻了BDE这边的发现？仔细想过，答案是不会，但要更谨慎**：隔壁的bug是"比较了两个
构造方式、大小都不同的测试集"；BDE这边的做法结构上不同——是把**同一个、大小固定**的
`molecule_cold_split`测试集，按"scaffold是否在train里出现过"做事后二分，两组都来自
同一个正确大小的测试集，不存在"测试集大小不匹配"这类bug的空间。而且**在B5(BonDNet)上
独立复现了同样方向的结果**(醛侧leaked-MAE 1.766 vs novel-MAE 2.744，+55%；产物侧3.596
vs 5.851，+63%——方向和B6一致，幅度略小但依然可观)。两个架构一致是个好迹象，但两次检查
用的是同一份分析脚本，这是"同一现象在两个模型上复现"，不完全等于"检验方法本身被独立
复现"。**结论**：把这个发现当作"大概率真实、但还没达到隔壁那种从头训练对照实验的确认
标准"来看待，不当作已经实锤的结论——真正严谨的下一步是**从头做一次scaffold-disjoint
划分再训练**，并且要吸取隔壁的教训：动手前先确认两边测试集规模真的可比。

### 3. 发现并修复的严重 bug

`calc_bde_alfabet.py` 最初版本:用**原始输入 SMILES**的原子顺序算出目标键的 `bond_index`,
却把它和 **RDKit 规范化后的 SMILES** 一起传给 `alfabet.model.predict`——而 ALFABET 内部
(`alfabet.fragment.Molecule.smiles` 属性)永远会重新规范化并用规范化顺序算自己的
`bond_index`。抽查发现 **98.9% 的源 SMILES 本来就不是 RDKit 规范形式**,所以绝大多数分子
被错误配对到了不相关的键(例如某个 id 的 formyl C-H 在原始顺序下是 bond_index 34,但在
规范化顺序下其实是 30,脚本因此取到了一个无关的芳环 C-H 预测)。

**已修复**:两个 `target_bond_*` 函数改为先规范化、再在规范化 mol 上算索引。用具体出错
分子(id 52677.0)验证过修复正确。**之前跑完的全库醛 ALFABET 结果已作废并重新生成**。

### 4. 第二个 bug(2026-07-15 凌晨发现,产物侧全库任务因此失败重跑)

产物侧修复版 ALFABET 全库任务(`24620818` array task 1)跑了 **6 小时 37 分钟**(片段枚举+
TF 预测两阶段都跑完了),最后在写出结果前于 ALFABET **自己的内部代码**里崩溃:
`pd.DataFrame(...).T.unstack().reindex(...)` 抛出 `ValueError: cannot handle a non-unique
multi-index`。根因:`calc_bde_alfabet.py` 把**未去重**的规范化 SMILES 列表传给
`alfabet.model.predict`——如果多个不同 id 的分子恰好规范化成同一个字符串(产物库里这种
重复比醛库更常见),之前的实现只是静默丢弃除最后一个 id 外的行(注释里也承认了这一点),
但对产物数据这批具体的重复模式,ALFABET 自己在 unstack 一个非唯一 MultiIndex 时会直接崩溃
而不是静默出错。

**已修复**(`pipeline/compute/calc_bde_alfabet.py`):调用 `predict` 前先对规范化 SMILES
去重,预测完成后用 `pandas.merge` 按 `(molecule, bond_index)` 关联回**每一个**原始 id——
这比修复前更正确:以前重复结构会丢行,现在所有共享同一结构的 id 都能拿到(相同的)正确
预测值。小样本验证过无回归。**产物侧任务已用修复后的脚本重新提交**(job `24625758`,
`--time=10:00:00`,预计仍需 6-7 小时量级,因为片段枚举+TF预测本身很慢,和是否去重关系
不大)。

### D-SPOC 醛/产物不对称的根因:找到了,不是"差分思路不适合醛侧"

用完整 217 个 RDKit 描述符重跑醛侧 D-SPOC(而非精选的 18 个)——见"三"节任务状态。同时
直接排查了此前 18 特征版本里最重要的几个特征,发现一个关键问题:**`MolWt` 在醛侧的
"差分"特征 `x(frag1)+x(frag2)-x(parent)` 对 500 个随机分子实测均值 = -6e-15、标准差 =
2.7e-14——数值上完全是零(浮点噪声)**,原因很简单:formyl C-H 均裂产生的两个碎片是
**极端不对称的一大一小**(酰基自由基 ≈ 母体去掉一个 H,单原子 H 自由基贡献几乎为零),
所以"两碎片描述符之和 - 母体描述符"对任何全局性描述符(分子量、TPSA、可旋转键数等)
都会精确抵消成一个（分子量场景下确实精确为)常数,不携带任何信息。但 XGBoost 的
`feature_importances_` 仍然给 MolWt 打了 0.227 的重要性分——这是**浮点舍入噪声恰好和分子
大小间接相关**造成的伪信号,不是真实化学信号。真正携带信号的是 `MaxPartialCharge`/
`MinPartialCharge`(合计 0.563),这两个是逐分子迭代计算的 Gasteiger 电荷,不会像
MolWt 一样精确抵消。

产物侧不存在这个问题,因为 ketC-carbC 断裂产生的两个碎片体量相当(都是分子的"一半"),
"两碎片之和减母体"是一个真正的两体比较,`MolLogP`(0.572)这类全局描述符能捕捉到两个
碎片相对的疏水性/共轭程度差异,这正是自由基稳定性的一个合理代理——所以 D-SPOC 在产物侧
表现好不是运气,是因为断裂方式本身就更适合"求和再相减"这种全局描述符的比较方式。

**推论**:醛侧 D-SPOC 表现差不是"差分表示不如图嵌入"这么简单的结论——它更准确的说法是
"对于一大一小的不对称碎片对,全局分子描述符的差分退化成了单碎片描述符,丢失了大部分设计
初衷的两体比较信息"。真正该用于醛侧的应该是**局部**描述符(比如具体在 formyl C/O 原子上
的电荷、Fukui 指数——仓库 `aldehydes_all.csv` 里已经算过 `fukui_plus_CHO_C` 等这类特征),
而不是全局分子级描述符的差分。这是一个有价值的后续改进方向,但今天没有时间重新实现。

- **确认支持局部(原子/键级)描述符定位**:读源码确认 `atom_lists`/`bond_lists` 输入参数
  真实存在且被使用(`qspoc/descriptor.py`),可以像 H-SPOC 那样精确指定 formyl C/H 原子或
  ketC-carbC 键的 RDKit 原子索引(复用 `calc_bde_alfabet.py` 里已有的 SMARTS 定位逻辑),

**未做的部分(需要你决定是否投入)**:全库规模(22 万×2 分子)跑一遍 Q-SPOC 需要对每个

**Pilot 结果(10 分子,`pipeline/bde/qspoc_pilot.py`)——结论:局部(atom/bond)描述符这条路
目前走不通,全局(cdft/esp)描述符可用**:

- `--include cdft,esp,atom,bond` 一起跑:CDFT 步骤对 4/10 分子报错(`N-1.wfn` 找不到、
